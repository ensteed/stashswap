#+DESCRIPTION: Yarn destashing marketplace — full project plan with phases and time estimates

* Overview

StashSwap is a marketplace for buying and selling partially used yarn skeins.
Stack: Node/Express/TypeScript · HTMX (server-side rendering) · MongoDB · AWS S3 · Stripe

** Current State (Done)
- JWT auth with bcrypt password hashing and HTTP-only cookie sessions
- User creation with full validation (username, email, password rules)
- Profile editing with picture upload (currently local filesystem — needs S3 migration)
- HTMX-driven UI with modal dialogs and conditional navbar

** Total Estimated Timeline
~18–26 weeks solo, assuming part-time work (~10–15 hrs/week).
Full-time (~40 hrs/week) cuts this to ~6–9 weeks.

* Phase 2 — Listings (Selling Side)
:PROPERTIES:
:ESTIMATE: 3–4 weeks
:END:

The core product of the platform. Nothing to buy until sellers can list.

** MongoDB Model (=listings= collection)
#+BEGIN_SRC javascript
{
  _id,
  seller_id,
  status,            // "draft" | "active" | "sold" | "cancelled"
  title,
  description,
  price,             // stored in cents
  condition,         // "new" | "like_new" | "good" | "fair"
  quantity,          // defaults to 1
  quantity_remaining,
  photos: [{ s3_key, order }],
  tags: {
    fiber_types: [],   // "merino", "alpaca", "acrylic", ...
    yarn_weights: [],  // "lace", "fingering", "DK", "worsted", "bulky", ...
    colors: []         // free-form tags
  },
  brand,             // e.g. "Malabrigo", "WoolPeople"
  weight_grams,      // grams remaining in the skein
  yardage_remaining, // yards/meters remaining
  ships_from_zip,    // for shipping quotes
  created_at,
  updated_at
}
#+END_SRC

* Phase 3 — Cart & Guest Checkout
:PROPERTIES:
:ESTIMATE: 2–3 weeks
:END:

The hardest UX problem: guests must be able to shop without creating an account.

** MongoDB Model (=carts= collection)
#+BEGIN_SRC javascript
{
  _id,
  session_id,       // for guests (stored in cookie)
  user_id,          // for logged-in users (nullable)
  items: [{
    listing_id,
    seller_id,
    quantity,
    price_snapshot, // price at time of add — lock it in
    title_snapshot  // in case listing is edited or deleted
  }],
  guest_email,      // collected at checkout start
  expires_at        // TTL index — auto-clean abandoned carts after 7 days
}
#+END_SRC

** Tasks
*** TODO Session cookie for guests
    - On first "Add to Cart", generate a UUID session ID and store in a cookie
    - Create or retrieve cart by session_id

*** TODO Add to Cart endpoint
    - Works for both guests and logged-in users
    - Validates listing is =active= and has available quantity
    - Snapshots price and title at add time

*** TODO Cart page
    - Review items, quantities, remove items
    - Group items by seller visually (one checkout creates one order per seller)
    - Show estimated totals

*** TODO Cart merge on login
    - When a guest logs in, merge their guest cart (by session_id) into their user cart
    - De-duplicate items, prefer higher quantity

*** TODO Handle multi-seller carts
    - Cart can contain items from multiple sellers
    - Checkout creates one Order per seller, one Stripe PaymentIntent per seller
    - Show seller groupings clearly in cart and checkout UI

** Notes
- Don't check inventory at add-to-cart time — check and reserve at payment time.
  Two people can have the same item in their cart; first to pay wins.
- Cart TTL index (=expires_at=) keeps MongoDB clean automatically.

* Phase 4 — Checkout & Orders
:PROPERTIES:
:ESTIMATE: 4–5 weeks
:END:

The core of the product. This is the hardest phase — Stripe webhooks, inventory conflicts,
guest linking, and email all converge here.

** MongoDB Model (=orders= collection)
#+BEGIN_SRC javascript
{
  _id,
  buyer_id,                  // nullable — null for guest orders
  buyer_email,
  guest_token,               // UUID — used to link guest order to a new account later
  seller_id,
  items: [{ listing_id, quantity, price_cents, title }],
  shipping_address: { street, city, state, zip, country },
  status,                    // "pending_payment" | "paid" | "shipped" | "delivered" | "cancelled" | "refunded"
  stripe_payment_intent_id,
  stripe_application_fee_cents,
  platform_fee_cents,
  shipping_cost_cents,
  tax_cents,
  total_cents,
  tracking_number,
  shipped_at,
  created_at,
  updated_at
}
#+END_SRC

** Tasks
*** TODO Checkout flow — step 1: shipping address
    - Pre-fill from user's default address if logged in
    - Guest enters email + address here (store =guest_email= on cart)

*** TODO Shipping cost calculation
    - Start with flat-rate tiers by weight (simplest, ship it fast)
    - Upgrade to EasyPost or Shippo API later for live rates
    - Seller's =ships_from_zip= + buyer's zip + item weight → rate

*** TODO Tax calculation
    - Use Stripe Tax (simplest — already in your Stripe account)
    - Alternative: TaxJar if you need more control

*** TODO Platform fee calculation
    - Decide your cut — e.g. 8% of item subtotal
    - Applied as =application_fee_amount= on the Stripe PaymentIntent (Stripe Connect)

*** TODO Stripe PaymentIntent creation
    - One PaymentIntent per seller-group in the cart
    - Include =application_fee_amount= for your platform cut
    - On the PaymentIntent =metadata=: store =order_id=, =buyer_email=

*** TODO Inventory reservation at payment time
    - Before confirming PaymentIntent, check =quantity_remaining= is sufficient
    - If not, abort checkout and notify buyer which item sold out

*** TODO Stripe webhook handler (=/webhooks/stripe=)
    - =payment_intent.succeeded= → mark order =paid=, decrement listing =quantity_remaining=,
      set listing to =sold= if =quantity_remaining= reaches 0, send emails
    - =payment_intent.payment_failed= → release reservation, notify buyer
    - =charge.dispute.created= → flag order, notify you (admin)
    - Verify webhook signature with Stripe's SDK

*** TODO Order confirmation email to buyer
    - Summary of items, total, shipping address
    - CTA: "Create an account to track your order" → links to registration pre-filled with their email
    - Include =guest_token= in the link if guest order

*** TODO Guest-to-account linking
    - Registration page accepts =?email=X&token=Y= query params from confirmation email
    - On account creation, find all orders where =guest_token= matches, set =buyer_id=
    - Token is single-use — clear it after linking

*** TODO Order detail page (buyer)
    - Status, items, shipping address, tracking number (when available)
    - Accessible to guest via =guest_token= in URL (no login required)

*** TODO Orders list page (buyer)
    - Currently stubbed — fill it in
    - Logged-in users see all orders linked to their account

** Notes
- Stripe webhooks are the source of truth for payment confirmation — never trust the client redirect.
- The guest linking flow via email token is the most nuanced UX in the whole project. Sketch it
  on paper before coding it.
- Stripe handles PCI compliance for card data — never store card numbers yourself.

* Things Not to Forget

| Topic                 | Detail                                                                               |
|-----------------------+--------------------------------------------------------------------------------------|
| Refund/dispute policy | Define before launch — Stripe has dispute handling but you need a written policy     |
| Listing photos        | Decide max count (suggest 8), plan S3 key structure now: =listings/{id}/{n}.webp=   |
| Yarn brand field      | Buyers search by brand (Malabrigo, WoolPeople, etc.) — add to listing model         |
| Skein weight (grams)  | Core destash info — "partial skein" means nothing without a gram weight              |
| Yardage remaining     | Equally important — "200 of 400 yards remaining"                                     |
| Ships from zip        | Needed for accurate shipping rate quotes per seller                                  |
| Prohibited items      | Basic Terms of Service / listing policy before accepting payments                    |
| Rate limiting         | Add to auth endpoints (=express-rate-limit=) before going live                      |
| HTTPS / prod config   | Ensure cookies use =secure: true= in prod, rotate JWT secret from the dev default   |
| Admin panel           | You will need this. Build something minimal in Phase 7 at the latest.               |
